# 智汇办公 AI 智能平台 — 项目规划与人员分工

---

## 一、产品大致规划与核心需求

### 1.1 产品定位

**智汇办公** 是一款面向企业内部的 AI 智能办公平台，以"知识管理 + AI 对话"为核心，帮助企业员工高效检索和利用内部知识资产。系统集成了 RAG（检索增强生成）、Agent（智能体）、LangGraph（图编排推理）三种 AI 推理模式，支持多模块知识分类、文档管理、联网搜索、文档生成等能力。

### 1.2 目标用户

| 角色 | 描述 |
|------|------|
| 管理员 | 管理知识库、用户、系统配置；创建共享知识库供全员使用 |
| 普通用户 | 使用 AI 对话进行知识检索、问答、文档生成；管理个人知识库 |

### 1.3 核心功能模块

| 模块 | 功能描述 | 优先级 |
|------|----------|--------|
| **规章制度** | 对应规章制度知识库，面向全员共享的企业制度类知识问答 | P0 |
| **产品技术** | 对应产品技术知识库，面向技术文档、产品规范等知识检索 | P0 |
| **行政服务** | 对应行政服务知识库，面向行政流程、办事指南等问答 | P0 |
| **自由问答** | 用户可选择通用知识库及个人知识库进行自由问答 | P0 |
| **知识库管理** | 创建/编辑/删除知识库，上传/管理文档，向量重建 | P0 |
| **文档管理** | 支持 PDF/Word/Excel/PPT/Markdown/TXT 等多格式上传与解析 | P0 |
| **AI 对话** | 支持 RAG / Agent / LangGraph 三种模式，流式输出，多轮对话 | P0 |
| **联网搜索** | 本地知识库不足时，可通过 Tavily/SearXNG 进行互联网检索 | P1 |
| **文档生成** | Agent 模式下可生成 Word/PDF 文档供下载 | P1 |
| **对话收藏** | 用户可收藏重要对话，便于后续查阅 | P2 |
| **对话日志** | 记录所有对话历史，管理员可查看全部，普通用户仅看自己 | P1 |
| **用户管理** | 管理员可增删改查用户，RBAC 角色权限控制 | P1 |
| **系统配置** | 管理员可配置模型参数、缓存策略等系统级设置 | P2 |
| **工作台** | 仪表盘展示知识库数量、文档数量、对话数、热门问题等统计 | P1 |
| **个人资料** | 用户可查看/修改个人信息（头像、密码等） | P2 |

### 1.4 核心需求清单

1. **用户认证与权限**：JWT 登录，RBAC 角色区分（管理员/普通用户），菜单动态显示
2. **知识库体系**：4 个内置锁定知识库（不可删除）+ 管理员共享库 + 用户个人库
3. **文档处理链路**：上传 → 解析（多格式） → 语义切分 → 向量化（Embedding） → 存入 OpenSearch
4. **RAG 检索问答**：查询重写 → 混合检索（向量 + BM25） → Rerank 重排序 → LLM 生成回答
5. **Agent 智能体**：工具调用（联网搜索、数学计算、文档生成、日期查询等），支持 DeepSeek DSML 与 OpenAI 标准格式
6. **LangGraph 编排**：多步骤任务拆解，状态管理，跨多轮对话执行
7. **流式输出**：后端 SSE 流式推送，前端实时渲染 AI 回答
8. **文档生成下载**：Agent 可调用工具生成 Word/PDF，前端 fetch + blob 触发下载
9. **对话持久化**：Redis 存储会话历史，PostgreSQL 存储对话日志
10. **统计与缓存**：Redis 缓存仪表盘统计数据、热门问题

## 二、技术难点与风险评估

### 2.1 技术架构

```
前端：Vue 3 + Vite + Element Plus + Pinia + Axios
后端：FastAPI + SQLAlchemy + PostgreSQL
AI/ML：LangChain + LangGraph + Ollama（本地 Embedding） + DeepSeek API（云端 LLM）
向量库：OpenSearch（向量 + BM25 混合检索）
缓存：Redis（会话存储 + 统计缓存 + 热门问题）
搜索：SearXNG / Tavily（联网搜索）
文档：python-docx / PyPDF2 / openpyxl / fpdf2
```

### 2.2 技术难点

| 编号 | 难点 | 说明 | 应对策略 |
|------|------|------|----------|
| T1 | **RAG 检索质量** | 中文文档召回率受切分策略、Embedding 模型质量影响大 | 采用混合检索（向量 + BM25）+ RRF 融合排序；Embedding 使用 qwen3-embedding:0.6b 本地模型；支持 Rerank 重排序 |
| T2 | **流式输出稳定性** | SSE 流式传输在代理/网关环境下易被缓冲，导致前端无法实时渲染 | 前端直连后端绕过 Vite 代理；使用 fetch + ReadableStream 手动解析 SSE；流式期间纯文本渲染，结束后切换 Markdown |
| T3 | **Agent 工具调用可靠性** | LLM 可能不按规定格式调用工具，或返回 DSML/XML 格式而非 JSON | 双格式解析（OpenAI tool_calls JSON + DeepSeek DSML XML）；后端增加兜底检测，自动补调工具 |
| T4 | **多格式文档解析** | PDF/Word/Excel/PPT 等格式差异大，解析质量参差不齐 | 统一 document_parser 模块封装各格式解析器；对 PDF 使用 PyPDF2，复杂 PDF 可接入 MinerU 服务 |
| T5 | **向量维度一致性** | Embedding 模型更换后维度不匹配会导致 OpenSearch 查询失败 | 配置 EMBED_DIMENSION 统一管理；维度不匹配时自动重建索引 |
| T6 | **多轮对话上下文** | 指代消解、省略补全影响检索准确度 | query_rewriter 模块利用 LLM 进行查询重写，处理指代和省略 |
| T7 | **并发与性能** | 大量用户同时对话时，LLM API 调用和向量检索可能成为瓶颈 | Redis 缓存热点数据；LLM 调用设置超时；统计接口使用缓存 |
| T8 | **中文文件名下载** | Content-Disposition 头中文编码在不同浏览器兼容性差 | 采用 RFC 5987 `filename*=UTF-8''` 编码方案 |

### 2.3 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| **DeepSeek API 不可用/限流** | 中 | 高 | 支持本地 Ollama 模型作为备选；前端可切换"云端 API / 本地模型" |
| **Ollama Embedding 模型缺失** | 低 | 高 | 部署前确认模型已安装（qwen3-embedding:0.6b）；配置文件中指定模型名 |
| **OpenSearch 索引损坏** | 低 | 高 | 支持维度不匹配时自动重建；文档支持重新生成向量 |
| **前端流式渲染兼容性** | 中 | 中 | 使用标准 Fetch API + ReadableStream；不支持时给出明确提示 |
| **文档解析失败** | 中 | 中 | 对异常文件做容错处理；记录解析失败日志；支持重新上传 |
| **Redis 服务中断** | 低 | 中 | 对话历史同时持久化到 PostgreSQL；缓存失效时降级直接查库 |
| **知识库数据量过大** | 中 | 中 | 合理设置 CHUNK_SIZE/CHUNK_OVERLAP；按知识库分索引管理 |
